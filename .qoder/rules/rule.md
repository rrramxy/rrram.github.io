# Qoder 规则配置

## 基础规则

- 所有代理应使用中文进行交互和输出，以保证中文用户的使用体验；
- 自主运行错误3次就不再测试，整理总结问题并输出；
- 编程语言尽量为Python和TypeScript，非必要不使用其他编程语言；

## 任务执行规则

- **对于常规任务（代码修改、文件创建/编辑、git提交、git推送等），直接执行无需确认**；
- **仅在以下危险操作时需要确认**：删除文件、强制推送(git push --force)、数据库删除操作、涉及金额的操作；
- 用户说“自主执行”、“直接做”、“不用问我”时，完全不需要任何确认；
- 发现问题时先修复，最后统一报告，不要每修复一个问题就询问；
- 批量任务一次性完成，不要分多次对话逐步确认；
- 对于项目检查、代码优化、部署发布等任务，自主完成所有步骤；

## 代码质量规范

- 所有代码必须包含适当的注释，解释复杂逻辑；
- 函数/方法长度不超过 50 行，超过需拆分；
- 变量命名使用有意义的名称，避免无意义缩写；
- 每次修改代码后检查语法错误和类型错误；
- 遵循项目已有的代码风格和格式规范；
- 优先使用函数式编程模式，减少副作用；

## 安全规范

- 禁止在代码中硬编码敏感信息（密码、API密钥、Token等）；
- 所有用户输入必须进行验证和清理；
- 使用参数化查询防止 SQL 注入；
- 敏感操作需要添加权限验证；
- 定期更新依赖包，修复已知安全漏洞；

## 测试规范

- 新功能必须配套单元测试；
- 测试覆盖率不低于 80%；
- 修改现有代码时更新相关测试；
- 关键业务逻辑需要编写集成测试；
- 测试用例应覆盖正常流程和异常流程；

## Git 提交规范

- 提交信息遵循 Conventional Commits 规范（feat/fix/docs/style/refactor/test/chore）；
- 每次提交只包含一个逻辑变更；
- 提交前确保代码通过所有测试；
- 提交信息使用中文描述，清晰说明变更内容；
- 重要功能开发使用特性分支；

## 文件组织规范

- 相关文件放在同一目录下，保持模块化结构；
- 单个文件职责单一，不超过 300 行；
- 公共资源（图片、字体等）存放在 assets 目录；
- 配置文件统一放在 config 目录；
- 按功能模块组织代码，而非按文件类型；

## 文档规范

- 新增功能时同步更新 README 和相关文档；
- API 变更时及时更新 API 文档；
- 复杂功能提供使用示例和说明；
- 代码注释使用中文，解释"为什么"而非"做什么"；
- 重要的架构决策记录在文档中；

## 性能规范

- 避免不必要的计算和重复查询；
- 大数据量操作考虑分页和懒加载；
- 图片等静态资源进行压缩和优化；
- 关键路径代码注意时间复杂度；

## 协作规范

- 代码变更需要团队成员 Review；
- 使用 Issue 跟踪问题和需求；
- 重要变更提前沟通，避免冲突；
- 保持代码可维护性，方便他人理解；
